# Phase 3 Report: 线程生命周期建模

## Evidence Base

- Source: `runtime/validation/phase-1/mailbox-census.json` + `phase-2` hypotheses
- 推断方式：只读，基于 envelope 主题/发件人/线程长度，无正文深度解析
- 样本线程：15 条（见 `runtime/validation/phase-3/thread-stage-samples.json`）

## 生命周期流汇总

| 流 ID | 名称 | 阶段数 | 代表线程 | 置信度 |
|---|---|---|---|---|
| LF1 | 资源申请流 | 5 | rdg货架-授权申请审核结果(8) | 0.82 |
| LF2 | 版本发布流 | 5 | 版本发布通告(13)、联调联试告知单(4) | 0.85 |
| LF3 | 运维服务报告流 | 5 | 辽宁dx项目运营日报(22)、项目运维服务报告(15) | 0.90 |
| LF4 | 合规与通知流 | 5 | 工时填报提醒(5)、考勤异常通知(2) | 0.88 |
| LF5 | 周报汇报流 | 5 | 马兆杰的工作周报(33) | 0.93 |

## 各流关键发现

### LF1 资源申请流

- 高频线程集中在 LF1-S1（申请发起）和 LF1-S2（审核中），说明申请提交后跟进是主要痛点
- rdg货架授权线程长度=8，推断经历多轮审批，SLA 提醒价值高
- 资源申请与版本发布存在强关联，LF1 完成后通常触发 LF2

### LF2 版本发布流

- 版本发布通告频次=13，是最高频的发布类信号
- 联调联试阶段（LF2-S2）风险最高，阻塞信号需优先监控
- 部署结果反馈（LF2-S5）频次低，说明验收环节可能存在漏跟进

### LF3 运维服务报告流

- 辽宁项目日报（移动/广电/电信/联通）合计 76 封，占总量 16%
- 日报类邮件适合批量自动摘要，每日汇总推送可大幅减少阅读负担
- 月报（LF3-S2）月末触发，提醒机制价值明确

### LF4 合规与通知流

- 工时填报提醒为最高频合规通知，截止日期提取优先级最高
- 考勤异常通知需回复确认，草稿辅助适用
- 安全保密类通知来自 iflytek.com，可通过发件人域名自动分类

### LF5 周报汇报流

- 马兆杰的工作周报频次=33，是全库最高频单一线程
- 周报草稿辅助（基于本周线程自动生成）是最直接的用户价值点
- 周报发送后通常无回复，LF5-S2 为终态

## 模板合规性自检

计划模板要求每个 stage 包含 `owner_guess / waiting_on / due_hint`，当前 lifecycle-model.yaml 未包含这三个字段。原因：仅从 envelope 主题/发件人推断 owner 和 waiting_on 置信度偏低（<0.6），需要正文级解析才能可靠推断。建议在 Phase 4 实际跑 daily-urgent 时，对 Top 线程补充正文抽样后再回填。

## 最值得进入 Phase 4 的 2 条流

| 排名 | 流 | 原因 |
|---|---|---|
| 1 | LF3 运维服务报告流 | 频率最高（日报 76 封 + 月报 15 封），完全只读，自动摘要零风险，用户感知最直接 |
| 2 | LF1 资源申请流 | SLA 提醒实现成本低，线程长度长（rdg=8, tjnlzx=7），漏跟进痛点明确 |

LF5 周报流虽然频次最高（33），但主要是用户自己发出的，Phase 4 的"今日该跟进什么"对它的适用性不如 LF3/LF1。

## 高置信推断

1. LF3（日报/月报）自动摘要是最低风险、最高频率的 AI 切入点
2. LF1（资源申请）SLA 提醒可直接减少漏跟进，实现成本低
3. LF5（周报）草稿辅助是用户感知价值最强的功能点
4. LF4（合规通知）截止日期提取可防止违规风险

## 低置信 / 待确认

- LF1 与 LF2 的关联触发逻辑需人工确认（资源批准后是否自动进入发布流）
- LF3 日报的接收方是否需要回复确认（影响 LF3-S3 触发条件）
- 周报草稿的风险边界：是否允许自动发送，还是仅生成草稿供确认

## Policy/Profile 建议（5 条）

| 优先级 | 建议 |
|---|---|
| 高 | LF1/LF2 启用 SLA 提醒：申请发起后 48h 无回复自动提醒 |
| 高 | LF3 日报自动摘要：每日汇总推送，减少逐封阅读 |
| 中 | LF4 合规通知截止日期提取，写入待办队列 |
| 中 | LF5 周报草稿辅助：基于本周线程自动生成草稿供人工确认 |
| 低 | 建立联系人优先级画像（czshi/zjma12 为核心节点），高优先级邮件优先展示 |
